이 문서는 JARVIS 프로젝트를 처음 시작한 시점부터 2026-03-22까지의 주요 개발 결정과 구조 변경 이유를 정리한 기록이다.

## 1. 프로젝트 시작 방향

- 프로젝트명은 최종적으로 `JARVIS`로 확정했다.
- 목표는 단순 챗봇이 아니라 다음 흐름을 갖는 승인 기반 에이전트 UI였다.
  - 사용자 지령 입력
  - 플랜 생성
  - 사용자 검토/승인
  - 실행 태스크 확정
  - 실행
  - 보고
- 초기 프로토타입은 정적 UI 중심으로 빠르게 시작했다.

## 7. MCP에 대한 중요한 판단

### 7-1. MCP registry는 capability source of truth

- MCP 목록은 단순 나열이 아니라 planner가 참조하는 실제 capability 기준점이어야 한다고 정리했다.
- registry는 별도 서버로 분리했다.

### 7-2. 가짜 MCP 제거

- `Planner MCP`는 실체가 없는 개념적 항목이었고, 실행 단계에 등장하면 구조가 깨진다고 판단했다.
- 따라서 `Planner MCP`는 제거했다.
- `Memory MCP`도 실제 MCP라기보다 내부 인프라에 가까웠기 때문에 제거했다.
- 현재 방향:
  - 실행 가능한 MCP만 registry에 남긴다.
  - planning-only 개념은 registry가 아니라 planner 축 내부에서 처리한다.

## 8. LLM 호출 구조 변화

초기에는 여러 경로에서 LLM을 직접 호출하거나 각 runtime이 자체 방식으로 모델을 다루는 문제가 있었다.

이를 정리하기 위해 다음 결정을 내렸다.

- 모든 LLM 호출은 `LLM Bridge`를 통하도록 재구성한다.
- OAuth / API Key / provider 선택 / OpenAI 호환 호출은 브리지 책임으로 모은다.
- runtime은 인증을 직접 다루지 않는다.

## 9. Deep Agents / Sequential Thinking 판단

### 9-1. Deep Agents는 planner 축에 적합

- Deep Agents는 planning, replanning, tool-aware orchestration에 강하다고 판단했다.
- 하지만 executor까지 맡기기보다 우선 `planner 책임`에 집중시키는 것이 맞다고 정리했다.

### 9-2. Sequential Thinking은 planner가 아니라 assist

- Sequential Thinking은 선택적 전략 보조로만 사용해야 한다고 결론냈다.
- 역할:
  - 의도 정리
  - 분기 비교
  - 리스크 점검
- 하지 말아야 할 일:
  - 최종 task 확정
  - 실행 계획의 마지막 책임

## 10. 최종 아키텍처 원칙 정리

2026-03-22 기준으로 가장 중요하게 확정한 원칙은 다음 문장이다.

> 생각은 PlannerRuntime, 실행은 ExecutorRuntime, 번역은 TaskCompiler, 가능성 판단은 CapabilityMap, 운영 가시성은 Structured Trace.

이를 더 풀면:

- `SequentialThinkingAssist`
  - 선택적 전략 보조
- `DeepAgentPlannerRuntime`
  - 최종 planner / replanner
- `FallbackPlannerRuntime`
  - planner fallback
- `TaskCompiler`
  - normalized plan -> executable tasks
- `StableExecutorRuntime`
  - 실행 / 로그 / evidence / report
- `CapabilityMapService`
  - registry -> planner-friendly capability layer
- `TraceLogger`
  - planner/executor/fallback 구조화 trace

## 11. Planner와 Executor 분리

초기에는 planning, tasking, execution, reporting이 같은 흐름 안에 뒤섞여 있었다.

이후 다음과 같이 분리했다.

- planner 축
  - `DeepAgentPlannerRuntime`
  - `FallbackPlannerRuntime`
  - `SequentialThinkingAssist`
- executor 축
  - `StableExecutorRuntime`
  - `ReportBuilder`
- 공통 인프라 축
  - `PlanNormalizer`
  - `TaskCompiler`
  - `CapabilityMapService`
  - `TraceLogger`

이 분리는 이 프로젝트에서 가장 중요한 구조 개선 중 하나였다.

## 12. Prompt 중심 개선 원칙

- 특정 사용자 의도를 코드에 계속 하드코딩하는 방향은 잘못이라고 여러 차례 확인했다.
- 특히 filesystem 관련 질의에서 다음 원칙을 세웠다.
  - planner가 더 잘 분해하게 prompt를 조정한다.
  - skill이나 executor에 의미 해석 규칙을 계속 추가하지 않는다.
  - 하드코딩은 최후의 수단이다.

즉:
- 복합 질의 분해 문제는 우선 `planner prompt`와 `planning discipline`에서 해결한다.

## 13. Filesystem 관련 교훈

- Filesystem MCP는 실제 호출되도록 연결했지만,
- 많은 실패 원인은 MCP 자체보다 planner의 decomposition 부족 때문이었다.
- 중요한 교훈:
  - skill은 결과 해석만 최소한 담당해야 한다.
  - “최신/오래된/개수” 같은 판단을 skill에 계속 넣으면 planner가 망가진다.
  - planner는 애초에 조회/필터/정렬/집계 task를 구분해서 내보내야 한다.

## 14. 보고 체계에 대한 결정

- 실행 로그만 보여주는 것은 불충분하다고 판단했다.
- 보고는 단순 실행 사실이 아니라 사용자의 질문에 대한 실제 결론을 말해야 한다.
- 따라서 다음 원칙을 세웠다.
  - workflow 카드로 보고를 남긴다.
  - 동시에 assistant message로 최종 답을 말한다.
  - 사용자가 다시 “그래서 결과는?”를 묻지 않아도 되게 만든다.

## 15. 타임라인 렌더링 정책

- 초기에는 프론트 상태로 워크플로우 카드를 강제로 아카이브하는 임시 처리도 있었다.
- 하지만 최종적으로는 이 방향이 맞다고 정리했다.
  - 모든 대화와 워크플로우 상태를 DB 이벤트로 저장한다.
  - 프론트는 `conversation_events`를 시간순으로만 렌더링한다.
  - 화면 반응성은 optimistic event로 보완한다.

현재 타임라인 원칙:
- 사용자 입력 직후 말풍선 즉시 표시
- 이후 DB 타임라인으로 동기화
- 이전 플랜/검토/실행/보고는 새 질문 아래 고정되지 않고 시간순으로 위로 올라감

## 16. 운영 가시성에 대한 결정

- planner와 executor 품질을 비교하려면 trace가 필수라고 정리했다.
- 그래서 structured trace를 응답뿐 아니라 DB에도 남기도록 했다.
- 주요 기록 대상:
  - 어떤 planner가 시도되었는가
  - fallback이 일어났는가
  - 어떤 task가 실행되었는가
  - 어떤 evidence가 수집되었는가
  - 최종 report가 무엇을 근거로 했는가

## 17. Python 환경 전환

- Deep Agents와 관련 라이브러리 요구사항 때문에 backend Python 환경을 `3.11` 기반으로 재구성했다.
- 방식:
  - `pyenv`
  - Python `3.11.x`
  - 새 `.venv`

## 19. 현재 프로젝트의 핵심 철학

현재까지의 결정들을 한 줄로 요약하면 다음과 같다.

> JARVIS는 단순한 규칙 땜질형 봇이 아니라, capability registry를 기준으로 Deep Agent가 실행 가능한 계획을 만들고, Stable Executor가 그것을 안정적으로 수행하며, 모든 과정이 trace와 report로 남는 승인 기반 에이전트 시스템이어야 한다.
