# JARVIS 요구사항 검토서

작성일: 2026-03-20

## 1. 검토 목적

본 문서는 지금까지 대화에서 정의된 JARVIS 요구사항을 정리하고, 실제 코드와 산출물, 실행 상태를 대조한 결과를 기록하기 위한 검토 산출물입니다.

검토 범위:
- 프론트엔드
- 메인 백엔드
- MCP Registry 서버
- Planner MCP 서버
- 로그인/세션
- Prompt 관리
- 저장소 구조와 운영 포트

## 2. 요구사항 목록

### 2.1 핵심 제품 요구사항

1. JARVIS는 챗봇 인터페이스여야 한다.
2. 사용자가 지령을 내리면 곧바로 실행하지 않고 먼저 플랜을 세워야 한다.
3. 플랜은 사용자 검토와 승인을 거쳐야 한다.
4. 승인 후 실행 태스크를 확정하고, 그 태스크 순서대로 수행해야 한다.
5. 플랜은 현재 사용 가능한 MCP를 기준으로 세워야 한다.
6. MCP 레지스트리는 별도 서버로 분리되어야 한다.
7. 메인 앱은 MCP 레지스트리 서버를 REST로 호출해 목록을 받아와야 한다.
8. Planner는 내부 개념이 아니라 독립 서비스여야 한다.
9. 프롬프트는 코드 상수가 아니라 별도 저장소에서 관리되어야 한다.
10. 프롬프트는 관리자 화면에서 생성/수정/삭제할 수 있어야 한다.
11. 프롬프트는 버전 관리와 활성 버전 복구가 가능해야 한다.
12. 저장은 SQLite 기반이어야 한다.

### 2.2 로그인/세션 요구사항

13. 로그인은 OpenAI GUI OAuth 방식이어야 한다.
14. OpenClaw/Codex 스타일로 로그인 및 세션 유지가 되어야 한다.
15. 로그인/로그아웃 버튼이 있어야 한다.
16. 로그아웃 후에는 세션이 해제되어야 한다.
17. 미로그인 상태에서는 질문 처리 기능이 막혀야 한다.

### 2.3 MCP/UI 요구사항

18. 오른쪽 화면에는 MCP 목록이 보여야 한다.
19. MCP를 클릭하면 상세 정보가 보여야 한다.
20. MCP 패널은 슬라이드처럼 열고 닫을 수 있어야 한다.
21. 레지스트리 관리 UI는 일반 사용자 화면과 분리되어야 한다.
22. 관리자 화면에서 MCP 활성/비활성화가 가능해야 한다.
23. 관리자 화면에서 바뀐 MCP 상태는 사용자 화면에 즉시 반영되어야 한다.

### 2.4 채팅 UX 요구사항

24. Python 백엔드, Svelte 프론트엔드여야 한다.
25. 멀티턴 대화 문맥이 반영되어야 한다.
26. 질문 전송 시 입력창은 비활성화되어야 한다.
27. 성공 시 입력창은 비워져야 한다.
28. 실패 시 입력 내용은 유지되어야 한다.
29. 응답 후 포커스는 다시 입력창으로 가야 한다.
30. 첫 진입/새로고침 시에도 입력창 포커스가 가야 한다.
31. `Cmd+Enter`로 전송 가능해야 한다.
32. 기본 브랜드명은 `JARVIS`여야 한다.
33. 디자인은 아이언맨 헬멧 내부 HUD 계열이어야 한다.

### 2.5 운영/문서 요구사항

34. `soul.md`로 자비스의 말투/정신상태를 제어할 수 있어야 한다.
35. 로그인 구현 방식은 개발 가이드 문서로 남아야 한다.
36. SQLite 테이블 정의 문서가 있어야 한다.
37. JARVIS 서비스 포트는 7000번대를 사용해야 한다.
38. 단, OpenAI/Codex 호환을 위한 loopback callback 예외는 문서화되어야 한다.

### 5.2 실제 실행 검증

확인한 포트:
- `7100` Registry 서버 리슨 중
- `7200` Planner MCP 서버 리슨 중
- `7300` Main Backend 서버 리슨 중
- `7400` Frontend 서버 리슨 중

실제 응답 확인:
- `GET http://127.0.0.1:7100/health` 정상
- `GET http://127.0.0.1:7200/health` 정상
- `GET http://127.0.0.1:7300/api/prompts` 정상
- `GET http://127.0.0.1:7300/api/mcps` 정상
- 세션 없는 `GET http://127.0.0.1:7300/api/auth/status` 응답: `authenticated=false`
- 세션 없는 `POST http://127.0.0.1:7300/api/chat` 응답: `401`

브라우저 검증:
- 메인 화면 `http://127.0.0.1:7400/` 렌더링 확인
- 관리자 화면 `http://127.0.0.1:7400/#/registry-admin` 렌더링 확인
- 관리자 화면에서 `MCP Registry`, `Prompt DB` 탭 노출 확인

브라우저 콘솔:
- 확인된 콘솔 에러는 `favicon.ico 404` 1건

## 6. 주요 결함 및 리스크

### 결함 1. 메인 백엔드 health 엔드포인트 부재

심각도: 중간

증상:
- `GET http://127.0.0.1:7300/health`가 `404 Not Found`

영향:
- 운영 점검, 헬스체크, 프로세스 모니터링에 불리함

판정:
- Registry/Planner는 health가 있으나 Main Backend는 빠져 있음

### 결함 2. 포트 정책 예외 존재

심각도: 낮음

내용:
- 전체 서비스는 7000번대 포트를 사용하지만, OpenAI/Codex 호환 callback은 `1455` 예외를 사용함

판정:
- 문서화는 되어 있으므로 정책 위반이 아니라 예외 관리 상태로 판단

### 결함 3. 관리자 화면 권한 보호 부재

심각도: 중간

내용:
- `#/registry-admin` 접근 자체에 별도 권한 확인이 없음
- MCP 관리와 Prompt DB 관리를 누구나 열 수 있는 구조임

영향:
- 단일 사용자 로컬 앱에선 치명적이지 않으나, 제품화 시 바로 문제됨

## 8. 최종 결론

현재 JARVIS는 "컨셉 검증용 프로토타입" 수준은 넘었고, 구조적으로는 상당히 맞게 만들어져 있습니다.

이전 검토에서 가장 컸던 인증 우회 문제는 수정됐습니다. 현재는 브라우저 세션에 로그인된 `profile_id`가 없으면 채팅이 `401`로 차단됩니다. 이제 남은 우선 보완점은 health 엔드포인트 추가, 관리자 접근 제어, 승인 후 실행부의 구체화입니다.
