# 2026.04.02 Dev History

이 문서는 2026-03-22 이후부터 2026-04-02까지의 주요 개발 결정과 구조 변경 이유를 정리한 기록이다.

## 1. 라우팅 아키텍처 전면 재설계

### 1-1. 기존 문제

- 이전에는 `question | command` 분기 기반의 단순 분류기가 메인 라우팅을 담당했다.
- intent classifier가 질문/명령을 1단계로 분류하고, 이후 처리가 각기 다른 분기로 흘렀다.
- 문제점:
  - MCP capability를 제대로 참조하지 않는 라우팅 판단
  - 사용자 의도의 미묘한 차이를 반영하지 못함
  - decline 경로의 부재

### 1-2. 새 라우팅 3단계 구조

기존 분류기를 대체해 다음 3단계 구조를 도입했다.

1. **Intent Adjudication** (`intent_router.py`)
   - 사용자 요청을 해석해 구조화된 판단 메모를 생성
   - `IntentAdjudication` 스키마: user_goal, task_nature(informational/retrieval/action/unsupported), preferred_handler(auto/llm/mcp), required_capabilities, safety 등
   - LLM 기반 판단을 우선하되, 실패 시 키워드 휴리스틱 fallback

2. **Capability Resolution** (`capability_resolver.py`)
   - 판단 메모 위에 MCP-aware capability probe를 한 번 더 수행
   - `CapabilityResolution` 스키마: mode(mcp_action/mcp_retrieval/llm_fallback/decline), missing_capabilities, selected_mcp_id 등
   - 최종 실행 계약을 확정

3. **Capability Router** (`capability_router.py`)
   - MCP id와 capability 별칭 매핑
   - 활성 MCP 기반 가용 capability 집합 관리
   - retrieval MCP 선택 로직

### 1-3. 새 라우팅의 핵심 원칙

- 정책상 금지면 `decline`
- MCP가 더 직접적으로 처리 가능하면 MCP 사용
- MCP가 없거나 적절하지 않으면 `llm_fallback`
- 사용자가 처리 수단을 명시하면 그 의도를 우선 반영
- capability가 없다는 사실 자체로 판단을 왜곡하지 않고, ideal handling 기준으로 먼저 adjudication

## 2. ToolAnswerRuntime 도입

### 2-1. 도입 배경

- 조회형 질문(retrieval)에 대해 planner → approval → execution 전체 워크플로우를 타는 것은 과도했다.
- 파일시스템 조회나 법령 검색처럼 단건 MCP 호출로 충분한 경우에 대한 경량 경로가 필요했다.

### 2-2. 구현

- `tool_answer_runtime.py` 신설
- `mcp_retrieval` 모드일 때 `ToolAnswerRuntime.answer_question()` 호출
- filesystem MCP: micro-planner로 적합한 조회 tool 선택 → MCP 호출 → evidence 기반 답변 생성
- korean_law MCP: 키워드 기반 tool 선택(search_all, search_precedents, search_interpretations) → MCP 호출 → 근거 기반 답변 생성
- planner/approval 과정 생략, 직접 답변

## 3. Decline Policy 도입

- `decline_policy.py` 신설
- `decline` 경로 시 일관된 사용자 안내 메시지 생성
- missing_capabilities가 있으면 어떤 capability가 없는지 명시
- safety 위반이면 정책 위반 사유 표시
- 모든 경우 대안/우회 절차 안내 포함

- Snapshot 기반 ref 자동 해결: planner가 ref를 모르는 경우 snapshot을 먼저 수집한 뒤 target text 기반으로 가장 적합한 ref를 자동 선택

## 5. RuntimeFactory 도입

- `runtime_factory.py` 신설
- 환경변수(`JARVIS_AGENT_RUNTIME`) 기반으로 planner/executor 구현체를 조합하는 책임을 한 곳으로 모음
- main API가 구체 구현체를 직접 알지 않도록 분리
- 지원 조합:
  - `classic` → `FallbackPlannerRuntime`
  - `deepagents` → `DeepAgentPlannerRuntime` + fallback
  - `sequential` → `DeepAgentPlannerRuntime` + `SequentialThinkingAssist` + fallback
- executor는 항상 `StableExecutorRuntime`

## 12. 이 기간의 핵심 결정 요약

> 라우팅은 단순 분류기가 아니라 판단 메모(adjudication) → capability 확인(resolution) → 실행 계약(routing) 3단계로 가야 하며, 조회형은 전체 워크플로우 없이 ToolAnswerRuntime으로 경량 처리하고, 지원 불가 시에는 일관된 decline 메시지를 사용자에게 보여야 한다.
