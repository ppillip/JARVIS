# JARVIS SQLite 테이블 정의서

이 문서는 JARVIS가 사용하는 SQLite 파일 DB의 현재 테이블 구조를 정리한 문서입니다.

- DB 파일 경로: `data/jarvis.db`
- 기준 소스: [`sqlite_store.py`](/Users/ppillip/Projects/NiceCodex/backend/app/sqlite_store.py)
- 용도:
  - MCP 레지스트리 저장
  - 프롬프트 저장 및 버전 관리
  - 워크플로우 실행 스냅샷 저장
  - workflow trace 저장
  - 대화 타임라인 저장

주의:

- 이 문서는 `create_tables()` 기준 현행 스키마를 설명합니다.
- 입력창 아래의 내부 상태 텍스트 같은 UI 전용 상태는 DB에 저장하지 않습니다.

## 1. mcp_registry

MCP 레지스트리 항목을 저장하는 테이블입니다.

### 목적

- 현재 시스템에 등록된 MCP 목록 저장
- MCP 활성/비활성 상태 관리
- MCP 메타데이터 보관

### 컬럼

| 컬럼명 | 타입 | NULL | 설명 |
|---|---|---|---|
| `id` | `TEXT` | NO | MCP 고유 ID, PK |
| `name` | `TEXT` | NO | MCP 표시 이름 |
| `scope` | `TEXT` | NO | MCP 범주 |
| `description` | `TEXT` | NO | MCP 설명 |
| `capabilities_json` | `TEXT` | NO | capabilities 배열을 JSON 문자열로 저장 |
| `expected_input` | `TEXT` | NO | 예상 입력 |
| `expected_output` | `TEXT` | NO | 예상 출력 |
| `source_url` | `TEXT` | YES | 참조 URL |
| `package_name` | `TEXT` | YES | 패키지 이름 |
| `transport` | `TEXT` | YES | transport 종류 |
| `auth_required` | `INTEGER` | NO | 인증 필요 여부, `0/1` |
| `risk_level` | `TEXT` | NO | 위험도 (`low`, `medium`, `high`) |
| `enabled` | `INTEGER` | NO | 활성 여부, `0/1` |
| `created_at` | `TEXT` | NO | 생성 시각, ISO8601 |
| `updated_at` | `TEXT` | NO | 수정 시각, ISO8601 |

### 키

- PK: `id`

## 2. prompt_definitions

프롬프트의 논리적 식별자와 현재 활성 버전을 저장하는 테이블입니다.

### 목적

- 프롬프트 ID 단위의 상위 엔티티 관리
- 어떤 버전이 현재 활성 상태인지 기록

### 컬럼

| 컬럼명 | 타입 | NULL | 설명 |
|---|---|---|---|
| `id` | `TEXT` | NO | 프롬프트 고유 ID, PK |
| `active_version` | `INTEGER` | NO | 현재 활성 버전 번호 |
| `created_at` | `TEXT` | NO | 프롬프트 생성 시각, ISO8601 |
| `updated_at` | `TEXT` | NO | 마지막 변경 시각, ISO8601 |

### 키

- PK: `id`

## 3. prompt_versions

프롬프트의 실제 버전 이력을 저장하는 테이블입니다.

### 목적

- 프롬프트 수정 이력 보관
- 이전 버전 복구 지원
- 활성 버전 전환 지원

### 컬럼

| 컬럼명 | 타입 | NULL | 설명 |
|---|---|---|---|
| `prompt_id` | `TEXT` | NO | 프롬프트 ID, FK |
| `version` | `INTEGER` | NO | 버전 번호 |
| `name` | `TEXT` | NO | 해당 버전의 이름 |
| `description` | `TEXT` | NO | 해당 버전의 설명 |
| `content` | `TEXT` | NO | 해당 버전의 프롬프트 본문 |
| `created_at` | `TEXT` | NO | 버전 생성 시각, ISO8601 |

### 키

- PK: `(prompt_id, version)`
- FK: `prompt_id -> prompt_definitions.id`
  - `ON DELETE CASCADE`

## 4. workflow_runs

플랜 생성부터 실행 보고까지 하나의 run 단위를 저장하는 테이블입니다.

### 목적

- 승인/실행 워크플로우의 최신 스냅샷 저장
- run 목록 조회
- run 상세 조회 시 plan/tasks/report 복원

### 컬럼

| 컬럼명 | 타입 | NULL | 설명 |
|---|---|---|---|
| `id` | `TEXT` | NO | run 고유 ID, PK |
| `command_text` | `TEXT` | YES | 원본 지령 |
| `phase` | `TEXT` | NO | 현재 단계 |
| `planner_type` | `TEXT` | YES | planner 종류 |
| `fallback_used` | `INTEGER` | NO | fallback 사용 여부, `0/1` |
| `plan_json` | `TEXT` | YES | plan snapshot JSON |
| `tasks_json` | `TEXT` | YES | task list JSON |
| `report_json` | `TEXT` | YES | execution report JSON |
| `created_at` | `TEXT` | NO | 생성 시각, ISO8601 |
| `updated_at` | `TEXT` | NO | 수정 시각, ISO8601 |

### 키

- PK: `id`

## 5. workflow_trace_events

개별 run에 대한 structured trace 이벤트를 순서대로 저장하는 테이블입니다.

### 목적

- planner/executor 동작 trace 저장
- run 상세 조회 시 trace 복원
- 운영 디버깅과 회귀 분석 지원

### 컬럼

| 컬럼명 | 타입 | NULL | 설명 |
|---|---|---|---|
| `id` | `INTEGER` | NO | 자동 증가 PK |
| `run_id` | `TEXT` | NO | run ID, FK |
| `sequence_no` | `INTEGER` | NO | run 내부 이벤트 순번 |
| `event` | `TEXT` | NO | trace 이벤트 이름 |
| `payload_json` | `TEXT` | NO | trace payload JSON |
| `created_at` | `TEXT` | NO | 생성 시각, ISO8601 |

### 키

- PK: `id`
- FK: `run_id -> workflow_runs.id`
  - `ON DELETE CASCADE`

## 6. conversation_events

채팅 세션 단위의 타임라인을 저장하는 테이블입니다.

### 목적

- 사용자 메시지, 답변, 시스템 메시지 저장
- 플랜/승인/실행/보고 snapshot 저장
- 좌측 대화 목록과 중앙 타임라인 복원

### 컬럼

| 컬럼명 | 타입 | NULL | 설명 |
|---|---|---|---|
| `id` | `INTEGER` | NO | 자동 증가 PK |
| `conversation_id` | `TEXT` | NO | 대화 세션 ID |
| `sequence_no` | `INTEGER` | NO | 세션 내부 이벤트 순번 |
| `event_type` | `TEXT` | NO | 이벤트 타입 |
| `payload_json` | `TEXT` | NO | 이벤트 payload JSON |
| `created_at` | `TEXT` | NO | 생성 시각, ISO8601 |

### 키

- PK: `id`

### 대표 event_type

- `user_message`
- `assistant_message`
- `system_message`
- `workflow_snapshot`

## 관계

- `prompt_definitions` 1 : N `prompt_versions`
- `workflow_runs` 1 : N `workflow_trace_events`
- `conversation_events`는 별도 FK 없이 `conversation_id` 단위로 세션을 구성

## 현재 사용 중인 프롬프트 조회 방식

예:

1. `prompt_definitions`에서 `id='planner_system'` 조회
2. `active_version` 확인
3. `prompt_versions`에서 `(prompt_id, version)` 일치 행 조회
4. 해당 `content`를 실제 LLM 호출에 사용

## 운영 메모

- 좌측 대화 목록 제목/미리보기는 `conversation_events`를 요약해서 생성합니다.
- `workflow_snapshot` payload에는 현재 plan, tasks, execution log, execution report, trace, current run id가 함께 들어갑니다.
- 입력창 아래의 내부 상태 텍스트는 현재 UI 상태 표시용이라 저장되지 않습니다.

## 비고

- 기존 JSON 파일:
  - `backend/app/mcp_registry.json`
  - `data/prompts.json`
- 위 두 파일은 초기 마이그레이션 소스로만 사용될 수 있으며, 현재 운영 저장소는 SQLite `data/jarvis.db`입니다.
- `workflow_runs`, `workflow_trace_events`, `conversation_events`는 현재 운영 중인 핵심 테이블입니다.
