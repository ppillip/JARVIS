### 3.자비스 메모리 구조는 GPT류와 같은 관점으로 구현하기

- 기본 철학은 GPT류처럼
  `히스토리 + 요약 + 사용자 프로필/선호 + 검색 주입`
  구조로 간다
- 단, 운영 방식은 Claude Code식 계층형 메모리 구조를 참고한다
  - global memory
  - project memory
  - user memory
  - session summary memory
  - important decision memory
- 모델이 스스로 장기 기억하는 구조가 아니라,
  메모리 저장소에 적재하고 필요 시 검색해서 현재 프롬프트에 다시 주입하는 방식으로 구현할 것
- Memory MCP처럼 포장하는 것보다, 우선 JARVIS 내부 `MemoryService`로 구현하는 방향을 우선 검토할 것
- 물리 저장 대상은 다음처럼 나눌 것
  - 현재 세션 대화 히스토리
  - 장기적으로 유지할 사용자 선호/프로필
  - 프로젝트 규칙/작업 규칙
  - 중요한 작업 결과 요약
  - 아키텍처 결정 및 반복되는 실패/성공 패턴
- 저장 기준
  - 앞으로도 답변/행동에 영향을 주는 정보
  - 사용자가 명시적으로 강조한 정보
  - 여러 작업에 반복적으로 등장한 정보
  - 프로젝트 전반 규칙으로 재사용될 수 있는 정보
  - 일회성 잡음이 아닌 정보
- 반복 판단 기준
  - 같은 선호/원칙이 여러 번 직접 언급됨
  - 서로 다른 작업에서 같은 요구가 반복됨
  - 시간이 지나도 계속 유지됨
  - 여러 run/trace에서 같은 패턴으로 나타남
- 업데이트 정책
  - 새 정보가 기존 정보보다 명확하고 최신이면 교체
  - 일부만 바뀌면 병합
  - 더 이상 유효하지 않으면 약화 또는 삭제
  - 충돌 시 최근성, 명시성, 반복성, 영향도를 함께 비교
- planner/executor와 메모리 계층은 분리할 것
  - planner는 관련 메모를 retrieval로 주입받아 사용
  - executor는 실행 결과에서 메모리 후보를 생성만 하고, 직접 장기 메모리 확정은 하지 않음
- 메모리는 “전부 저장”이 아니라 “후에 다시 쓸 가치가 있는 정보만 저장” 원칙으로 설계할 것
- 메모리를 읽는 시점
  - 사용자 요청이 들어와 planner가 시작되기 직전
  - request classifier 이후, deep agent planner 이전
  - 필요 시 sequential thinking assist에도 같은 메모리를 주입
- 메모리를 주로 참조하는 계층
  - `DeepAgentPlannerRuntime`
  - 필요 시 `SequentialThinkingAssist`
  - executor는 원칙적으로 장기 메모리를 직접 참조하지 않음
- 메모리를 저장/갱신하는 시점
  - 실행 후
  - 보고 후
  - 사용자 선호나 프로젝트 규칙이 명시적으로 드러난 직후
- 메모리 저장 책임
  - executor/report 계층은 메모리 후보만 생성
  - 최종 저장/병합/교체/삭제 판단은 `MemoryService`가 담당
- planner metadata 또는 structured trace에는
  - 어떤 메모리를 참조했는지
  - 어떤 규칙이 플랜에 영향을 줬는지
  남길 수 있게 설계할 것
